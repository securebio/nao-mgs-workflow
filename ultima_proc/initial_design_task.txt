From the core
"The yield of this UG100 sequencing run was decent, at 9.015 billion reads. The library was indexed with a single Ultima barcode resulting in a single unaligned CRAM output file containing all of the reads for the library. I then used the Ultima "Trimmer" tool to demultiplex the reads using the sample barcode information you provided in the sample submission form. The Trimmer tool searches for and removes the Illumina adapter sequences at the beginning and ends of the reads, extracting the i5 and i7 index sequences from them, and demultiplexes the reads based on those sequences.

For relatively short  (~200bp or smaller) library fragments the Illumina adapters with indexes will be seen at both ends of the read. For longer library fragments only the adapter sequence (and its i5 index sequence) at the beginning (5') end of the read will be seen. Since these samples were indexed with dual unique barcodes it is only necessary to see one of the indexes in order to assign a read to a sample. The Trimmer tool outputs two files for each sample, one with reads where both the i5 and i7 index sequences were seen (e.g. 44_20250211-ATTAACAAGG-TGTTGTTCGT.cram), and one with reads where only the i5 index was seen (e.g. 44_20250211-NA-TGTTGTTCGT.cram).

You'll likely want to convert these unaligned CRAM files to fastq format, which can be done with samtools:
samtools fastq -0 output.fastq input.cram
Then for each sample you could concatenate the two fastq files into a single fastq file and then proceed with processing the data such as running FastQC or aligning to a reference genome with BWA or Bowtie2.

So, to summarize:
The reads are adapter trimmed
Our assessment of what appeared to be i5 and i7 reads is correct
You were correct that the "NA-TGTTGTTCGT"-style files contain reads where it didn't encounter the i7 index
It's recommended that we concatenate the two CRAMs/FASTQs for each sample"
